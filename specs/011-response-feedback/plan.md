## Implementation Phases

### Phase A вЂ” Backend (independent)

1. **Add `ResponseFeedback` model** to `backend/models/db.py`
   - Fields: `id`, `log_id` (FK, UNIQUE), `is_positive`, `comment`, `created_at`, `updated_at`
   - SQLAlchemy `create_all` auto-creates table on startup

2. **Create `backend/api/v1/feedback.py`** router
   - `POST /api/v1/feedback` with upsert semantics (ON CONFLICT DO UPDATE via SQLAlchemy merge)
   - Validate: `log_id` exists (404 if not), `comment` в‰¤ 500 chars (422 if not)
   - Register router in `backend/api/__init__.py`

3. **Extend `GET /logs`** in `backend/api/v1/logs.py`
   - LEFT JOIN `response_feedback` on `log_id`
   - Add `feedback: {is_positive, comment, updated_at} | null` to each log entry

4. **Write backend tests** in `backend/tests/test_api.py`
   - POST creates feedback
   - POST updates existing feedback (upsert)
   - POST returns 404 for unknown log_id
   - POST returns 422 for comment > 500 chars
   - GET /logs includes `feedback` field (null and non-null)

### Phase B вЂ” Frontend (depends on Phase A API)

5. **Extend `rag-api.service.ts`**
   - Add `FeedbackEntry` interface (`is_positive`, `comment`, `updated_at`)
   - Extend `LogEntry` with `feedback: FeedbackEntry | null`
   - Add `submitFeedback(logId, isPositive, comment?)` в†’ `Observable<FeedbackEntry>`

6. **Add feedback UI to chat**
   - Track `feedbackState` per message: `{ logId, isPositive: boolean | null, comment: string, showComment: boolean, submitting: boolean, error: string | null }`
   - Show buttons only when `type === 'done'` event received from stream
   - рџ‘Ќ click: set `isPositive = true`, hide comment, call API
   - рџ‘Ћ click: set `isPositive = false`, show comment textarea
   - Submit button: call API with comment, then hide textarea
   - Handle network errors inline on the button

7. **Extend logs view**
   - Add `Feedback` column to the logs table
   - Display рџ‘Ќ / рџ‘Ћ / вЂ” based on `feedback` value
   - Show comment in tooltip or expanded row if present

8. **Write frontend tests**
   - Chat: buttons appear after `done` event; рџ‘Ќ hides comment; рџ‘Ћ shows comment; submit calls service
   - Logs: renders рџ‘Ќ, рџ‘Ћ, and null feedback correctly

## Key Design Decisions

See `research.md` for full rationale. Summary:

- **Upsert** on `POST /feedback` (no separate PATCH) вЂ” simplifies client state
- **Embed feedback** in `GET /logs` response (JOIN) вЂ” avoids N+1
- **Boolean `is_positive`** вЂ” matches binary spec, simple to index
- **Separate `response_feedback` table** вЂ” keeps `QueryLog` focused on traceability
- **Inline comment textarea** вЂ” standard UX pattern, no modal
